# Шаг 0012: Понимание `ownerId` и `owner` на примере `User -> Channel`

## Команды терминала

```bash
npx prisma validate
```

## Конспект шага

Разобрали смысл двух полей в relation:

- `ownerId` - это техническое поле внешнего ключа (FK), которое реально хранится в таблице БД.
- `owner` - это объектная ссылка Prisma на связанную сущность, чтобы удобно работать в коде.

Пример из `Channel`:

```prisma
ownerId String @map("owner_id")
owner   User   @relation(fields: [ownerId], references: [id])
```

Читается так:

- в таблице `channels` есть колонка `owner_id`,
- ее значение должно совпадать с `users.id`,
- в Prisma-коде можно обращаться к владельцу канала через поле `owner`.

Обратная сторона связи в `User`:

```prisma
channels Channel[]
```

Это список всех каналов, где `ownerId` ссылается на этого пользователя.

## Та же логика для `Channel -> Video`

Если видео принадлежит каналу, то структура аналогична:

- в `Video` есть FK-поле (техническое),
- рядом relation-поле на модель `Channel`,
- в `Channel` есть массив `Video[]` как обратная сторона.

Важно: названия полей должны отражать смысл.
`ownerId/owner` технически работает, но для связки `Video -> Channel` читаемее обычно `channelId/channel`.

## Что важно запомнить

- `...Id` = "что хранится в базе" (внешний ключ).
- relation-поле (`owner`, `channel`, `user`) = "как удобно ходить по связям в Prisma".
- Для связи `1 -> N` нужны обе стороны:
  у стороны `N` FK + `@relation`,
  у стороны `1` массив связанных сущностей.
- Если Prisma ругается на relation, чаще всего не хватает обратного поля или неверно указаны `fields/references`.

## Что делать дальше

- Для каждого нового relation проговаривать вслух:
  "какая колонка FK хранится в таблице" и "какой объект хочу получать в коде".
- После изменений схемы запускать `npx prisma validate`.
- Затем делать миграцию только после того, как модель читается по смыслу.
